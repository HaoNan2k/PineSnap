# Demo Case 设计文档：你的密码，没人知道

> 日期：2026-04-17
> 状态：已实现（v1）
> 路由：`/demo`

## 为什么做这个 Demo

PineSnap 需要一个可以同时用于三个场景的 demo case：

1. **Landing page 自助体验** — 用户无需注册就能体验完整学习流程
2. **社交媒体引流** — 截图/录屏有传播力
3. **投资人路演** — 3 分钟内证明"体验驱动学习"的价值

核心目标：让用户体验后留下两个感受——"原来我能懂"和"这比看文章有意思多了"。

## 话题选择

我们评估了三个候选方向：

| 候选 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| 函数式编程 | 技术深度好 | 受众太窄，投资人/普通人看不懂 | 否 |
| AI 原理科普 | 时效性强、流量大 | 容易变成"又一个 ChatGPT 解释 AI"，和竞品差异感弱 | 备选 |
| 密码哈希 | 每个人都有经验、反直觉惊喜强 | 需要把技术概念包装得平易近人 | **选定** |

**选密码哈希的核心理由：** 每个人每天都在经历"忘记密码 → 重置"这件事，但从来不知道背后的原因。这种"日常谜题"最容易制造"原来我能懂"的冲击力。而且"网站不知道你的密码"这个事实对大部分人是反直觉的，天然有传播力。

## A2UI 架构决策：Headless Component

在设计 demo 的过程中，我们确认了 A2UI 的核心架构理念：**交互原语与视觉表现分离**。

```
┌─────────────────────────────────────┐
│  视觉层（无限变化）                    │
│  每个 case 定制不同的视觉表现           │
├─────────────────────────────────────┤
│  交互原语层（少量、稳定）               │
│  SingleChoice / MultipleChoice /     │
│  FillInBlank / SocraticBranch        │
├─────────────────────────────────────┤
│  数据契约层（AI 只看这层）              │
│  { type: "single_choice",            │
│    selected: "B" }                   │
└─────────────────────────────────────┘
```

**为什么这样设计：**
- AI 模型只需要知道 4 种 tool schema，不需要为每种视觉变体训练
- 同一个 SingleChoice 原语可以长成完全不同的样子（按钮组、图片卡、左右滑等）
- 新的视觉样式不需要改 AI 工具定义
- 这比 Brilliant（组件和视觉耦合）和 Duolingo（固定交互形态）都更灵活

**原语扩展原则：** 只在数据契约真正不同时才加新原语（如 Ordering 排序、Slider 连续值）。纯视觉变化不加新原语。

## 工程架构：独立 Demo 页面

我们评估了三种方案：

| 方案 | 描述 | 优劣 |
|------|------|------|
| A: 独立页面 | `/demo` 新路由，复用 A2UI 组件，不走 AI 流水线 | **选定**。最干净，零修改现有代码 |
| B: 改造现有 Canvas | 给 LearningCanvas 加 demo mode flag | 给现有组件加分支逻辑，增加复杂度 |
| C: AI 约束模式 | 用特制 prompt 让 AI 生成确定性内容 | 不 100% 确定性，无法做自定义动画 |

选 A 的理由：demo 需要自定义动画（哈希变换、登录模拟等），这些远超 markdown 渲染能力。独立页面可以直接写 React 组件，同时复用 A2UI 交互原语保证一致性。5 个新文件，0 个修改现有文件。

## 交互设计决策

### 1. 开场引导：自动动画 + 用户关键点击

**决策：** 不让用户从头操作登录流程（太长），也不纯看动画（太被动）。用自动动画建立场景（输密码 → 登录 → 密码错误），然后在"忘记密码？"处留一个用户交互点。

**为什么：** 自动动画 3 秒建立场景够了。但"忘记密码"这个动作每个人都做过，让用户亲手点一次会唤起真实记忆，比被动观看更有共鸣。

### 2. 视觉引导：Dim + Hotspot，不用文字

**决策：** 密码报错后，整个登录表单 opacity 降到 0.35，只有"忘记密码？"保持不透明 + 绿色脉冲点。

**为什么：** 我们评估了四种引导模式：

| 模式 | 评估 |
|------|------|
| Spotlight 遮罩 | 太重，像企业 SaaS |
| Coachmark + 箭头 | 有"教程感"，和沉浸学习调性冲突 |
| 文字指令（"试试点击..."） | 太生硬，破坏体验 |
| **Dim + Hotspot** | **最克制**，不打断体验，好奇心驱动 |

### 3. 模拟 UI 要真实，引导文字放外面

**决策：** 登录框、重置密码表单等模拟 UI 的内容尽量还原真实界面。教学性的文字（"为什么只能重新设？"）放在模拟 UI 卡片外面。

**为什么：** 早期版本把"注意：我们无法告诉你旧密码"放在了重置密码卡片里面，用户反馈"很出戏"。模拟 UI 的沉浸感来自真实感，一旦混入教学文字就破坏了。

### 4. Continue = 交互完成（不管对错）

**决策：** 答错选择题后，显示正确答案反馈，Continue 按钮立刻可用。不卡用户。

**为什么：** demo 的目标是让人**完成全程并学到东西**，不是考试。答错本身就是学习，正确答案的揭示就是教学。卡住用户会让情绪从"有趣"变成"烦躁"，直接损害完成率。

### 5. 最后一步用叙事动画，不用大段文字

**决策：** 最后一步是一个自动播放的垂直流动画（密码 → 哈希 → 存储 → 原始密码消失），而不是文字总结。

**为什么：** 早期版本有一个"反思页"（大段文字回顾），用户反馈不需要。最后一页的动画已经是最好的总结——用视觉重演整个过程比文字复述更有力。amber 色标签"已理解：密码哈希"作为整个体验的情感收束。

## 5 步流程总览

| 步骤 | 交互原语 | 内容 |
|------|---------|------|
| 1 | SingleChoice | 自动动画登录 → 点忘记密码 → 重置密码界面 → "为什么只能重置？"四选一 |
| 2 | Canvas（无交互原语） | 哈希函数动画 + 雪崩效应对比 |
| 3 | SingleChoice | 数据库可视化 + "服务器存的是什么？"四选一 |
| 4 | SingleChoice | "登录时怎么验证？"三选一 |
| 5 | Canvas（叙事动画） | 密码 → 哈希 → 存储 → 消失 → amber 发现标签 |
