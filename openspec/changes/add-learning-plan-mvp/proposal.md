# Proposal: Learning Plan MVP（文本计划 + 右侧交互）

## Why

当前学习体验以“单卡片生成”为主，缺少对学习目标的澄清与可执行计划。MVP 需要先跑通“澄清 → 计划 → 互动”的流程，并与现有 Chat 模块解耦，避免历史/Sidebar/会话深链等复杂约束干扰学习体验验证。

## What Changes

- 引入学习会话流程，与现有 Chat Conversation 解耦
- 在学习入口新增“澄清 → Plan”阶段（纯文本）
- 交互区采用“右侧文本 + 输入框”的最小形态
- MVP 仅内存态存储，不落库，不引入 schema 变更
- MVP 阶段允许非流式模型调用

## 背景与动机

本项目希望将学习流程抽象为类似“Plan 模式”的体验：先明确学习目标，再进入交互推进。MVP 的目标不是做复杂题型，而是验证用户对“Plan + 互动”的接受度，因此以 Markdown 文本为输出载体即可满足早期需求。

## 范围（MVP）

### 包含

- **学习入口与 ID 体系**：
  - `/learn/[resourceId]` 作为学习入口
  - `resourceId` 用于关联澄清、Plan 与后续互动（MVP）
- **流程**：
  - Start → 生成澄清问题（3 题）
  - 提交答案 → 生成 Plan（纯文本）
  - Plan 之后进入文本交互（右侧交互区）
- **呈现**：
  - 右侧交互区展示 Markdown 文本
  - 交互区包含输入框

### 不包含（明确非目标）

- 不与现有 Chat Conversation/Sidebar/历史列表对接
- 不引入结构化题型 UI（仅文本）
- 不实现流式输出（MVP 可先非流式）
- 不落库，不新增 Prisma schema

## 风险与权衡

- **无持久化**：MVP 仅内存态，刷新即丢失；但换来更快的验证速度
- **文本主导**：缺少结构化 UI，交互效率较低，但更利于快速迭代

## 验收标准（MVP）

- 存在独立的学习会话入口 `/learn/[learningSessionId]`
- Start 后生成 3 个澄清问题（文本）
- 澄清完成后生成 Plan（纯文本）
- Plan 后进入右侧交互区，支持文本输入与回复
- 未引入任何数据库变更
