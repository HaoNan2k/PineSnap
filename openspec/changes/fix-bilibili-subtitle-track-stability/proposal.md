# 提案：修复 Bilibili 字幕错轨的扩展端稳定性防护

## 背景

在 Bilibili 扩展采集链路中，已复现“同一 `bvid/aid/cid` 下，`x/player/v2` 返回的 `ai-zh` 字幕轨不稳定”问题。  
当前实现采用“单次读取轨道并直接拉取 `subtitle_url`”策略，导致上游错轨被直接入库。

对应问题沉淀见：`docs/bilibili-subtitle-track-mismatch-bug-report.md`。

## 本次目标

1. 在不改动服务端契约的前提下，为扩展端主 extractor 增加“轨道稳定性防护层”。
2. 当上游轨道不稳定时，优先“保守失败”而非写入错误字幕。
3. 将轨道采样与判定过程写入 `metadata.captureDiagnostics`，便于后续排障与数据分析。
4. 在 `docs/` 增加可执行说明，描述策略、取舍与验证方法。

## 非目标

- 不引入新的服务端 API。
- 不修改 `POST /api/capture/jobs` 成功响应契约。
- 不引入模型级语义校验（如 LLM 判题）作为本次必选步骤。

## 决策清单

### 沿用既有决策

- 默认主路径仍为 `bilibili_full_subtitle_v1`。
- 服务端继续原样持久化 payload。
- 客户端继续通过 `x/player/v2` + `subtitle_url` 获取字幕正文。

### 本次新增/变更决策

- 将“单次选轨”升级为“多次采样 + 一致性判定 + 上限重试”。
- 当轨道样本无法形成稳定结论时，返回明确失败码（不入库错内容）。
- 新增诊断字段记录轨道样本、稳定性判定与重试过程。

## 预期收益

- 显著降低“错视频字幕入库”的概率。
- 保持主流程改动范围可控（局部封装，不侵入上传/鉴权链路）。
- 提升问题可观测性，便于长期优化和回归验证。
