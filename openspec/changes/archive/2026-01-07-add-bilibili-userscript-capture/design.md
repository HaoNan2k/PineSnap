# 设计：B 站 Userscript 采集（MVP）

## 设计原则

- **最小可用**：只覆盖“用户点击发送 → PineSnap 收到采集 payload”这一条路径。
- **低风险边界**：不涉及媒体下载与转写，不将系统定位为下载器。
- **可排障**：采集链路必须携带最小诊断信息（版本、extractor、URL）。
- **可扩展**：payload 为最小结构化形式，允许未来在不破坏兼容的前提下扩展字段。

## 交付方式（Userscript 如何给到用户）

MVP 推荐两种交付方式（按优先级）：

1) **GreasyFork 发布**

- 优点：用户安装体验好；脚本更新分发成熟；版本历史可追溯。
- PineSnap 需要提供：安装指引页（仅文档/引导），并在产品内提供“安装脚本”入口。

2) **PineSnap 自托管安装链接**

- PineSnap 提供一个静态可访问的 `*.user.js` 链接，用户在安装了脚本管理器后点击即可安装。
- 优点：不依赖第三方平台；缺点：更新与发现能力弱于 GreasyFork。

## 采集方式（基于 AI 小助手面板 DOM）

脚本通过 B 站「视频 AI 小助手」面板提取内容，包括：

- **视频总结**：总结正文 + 要点（时间点 + 文本）
- **字幕列表**：逐条字幕（时间戳 + 文本）

- 优点：不依赖不稳定/需要登录态的内部 API；对用户来说“所见即所得”。
- 缺点：对 DOM 结构/类名变化敏感；因此需要版本控制与可观测字段用于排障。

## 鉴权与跨域（仅定义策略，不落实现）

Userscript 运行在 `bilibili.com` 域名下，向 PineSnap 域名发送请求。为保证“用户点击发送”在跨域下可工作，推荐 MVP 使用“显式 token”方式：

- PineSnap 提供 **Capture Token**（与用户账户绑定，可撤销与轮换）。
- Userscript 在首次使用时引导用户粘贴 token（或从 PineSnap 页面复制），后续保存在脚本管理器的存储中。
- 请求发送时通过 `Authorization: Bearer <token>` 传递（payload 本身不携带鉴权信息）。

该策略的优点：

- 不依赖跨站 Cookie；
- 可在用户明确授权后工作；
- 容易做撤销、轮换与审计。

## 服务端端点契约（MVP）

服务端应提供一个稳定端点用于接收采集结果：

- `POST /api/capture/bilibili`
- Request body：`PineSnapBilibiliCapturePayloadMVP`
- Response（MVP）：返回一个确认结果（例如 `{ ok: true }`），并可附带一个 `captureId` 用于追踪。

## 失败与提示（MVP 交互）

脚本在失败时必须给出“可操作”的提示，至少区分：

- 未检测到字幕：提示用户检查“字幕是否存在/AI 小助手字幕是否可用”。
- 采集失败：提示用户刷新页面重试，并附带脚本版本以便上报。
- 发送失败：提示用户检查 token 是否配置/是否过期，或 PineSnap 服务是否可达。
