# 提案：引入 Chrome 扩展全量字幕采集（AI 面板仅保留代码，不参与主链路）

## 背景

当前 B 站采集主要依赖“视频 AI 小助手面板”。该路径在可用性与稳定性上存在外部依赖风险，且难以保证“一次触发即拿到全量字幕”。

同时，项目已明确后续方向：

- 采集能力优先落在用户客户端（浏览器扩展）；
- 服务端只接收结构化结果并入库；
- 不留存原视频/原音频；
- 既有数据协议与入库契约尽量保持兼容。

## 本次目标

1. 新增“Chrome 扩展 + 全量字幕优先”的 B 站采集能力，支持一次触发完成字幕抓取与发送。
2. 保持现有 payload key 与服务端响应契约稳定，尤其是成功响应中的 `resourceId` 形态不变。
3. 保留既有 AI 小助手面板 extractor 代码路径（不删除），但默认不参与应用层采集策略链路。
4. 采用可扩展、无冗余冲突的 extractor 架构，便于后续接入 YouTube 等平台。
5. 将“Chrome 扩展开发与发布前验证流程”沉淀到 `docs/` 文档，作为实现与验收标准的一部分。

## 非目标

- 不在本变更中实现 YouTube/公众号/小红书/抖音采集。
- 不在本变更中引入媒体下载或 ASR 转录链路。
- 不在本变更中改变 `Resource` 表结构、`resourceId` 生成机制与会话创建规则。
- 不在本变更中完成 Chrome Web Store 正式上架（先完成可本地加载与可回归验证的交付）。

## 决策清单

### 沿用既有决策

- 采集入口继续使用 `POST /api/capture/bilibili`，并使用 Capture Token 鉴权。
- 采集结果继续以结构化 JSON 原样写入 `Resource.content`（jsonb）。
- 采集与会话解耦：采集阶段不创建 Conversation/Message。
- 成功响应维持 `{ ok: true, resourceId: string }`。

### 本次新增/变更决策

- 客户端形态新增为 Chrome 扩展，采集逻辑以“全量字幕优先”策略执行。
- extractor 采用“单主路径执行 + 历史代码保留”链路：
  - 主路径：页面内嵌上下文 + 字幕 API/字幕轨抓取全量字幕；
  - 历史保留：既有 AI 小助手面板 extractor 文件保留但默认不进入执行链。
- 扩展向 `api.bilibili.com` 请求字幕相关 API 时，使用浏览器登录态凭据，以避免“网页已登录但接口视角未登录”。
- `cid` 解析采用“主来源 + 可验证反查兜底”策略，严禁伪造字段值。
- 引入 extractor provider 标识约束，保证不同分支结果可观测、可回溯、可灰度。

## 预期收益

- 显著降低对单一页面面板 DOM 的依赖，提升采集成功率与稳定性。
- 保持既有数据契约稳定，避免对资源存储与下游学习链路造成破坏性影响。
- 为后续多平台采集复用统一 extractor 架构与观测体系。
- 将实现步骤与验证清单文档化，降低后续接手与发布风险。
